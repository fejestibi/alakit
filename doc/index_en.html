1. Introduction

Alakit is a hybrid desktop application framework that combines the flexibility of modern web technologies with the safety and performance of Rust. It's not just "another Tauri-like" tool, but a fundamentally different approach to portable software development.

What is Alakit?

Alakit offers a middle ground: use pure HTML/CSS for the interface, but forget about JavaScript coding for the vast majority of logic.

Core Principles

  • No-JS (Optional): 90% of business logic can be written in Rust without writing a single line of JavaScript.
  • Encrypted Backend Store: Sensitive data is protected by AES-256 encryption, memory locking (mlock), and automatic memory clearing (zeroize) in Rust-side RAM (data is plaintext in WebView).
  • Auto-discovery: The framework automatically finds and registers your controllers thanks to macros.

2. Quickstart (The 5-minute Setup)

Let's get started with Alakit! We recommend a Workspace-based structure for clean separation.

Installation

Add the necessary dependencies to your Cargo.toml file:

[dependencies]
alakit = { path = "../alakit" }
alakit_macro = { path = "../alakit_macro" }
rust-embed = "8.11"

Hello World

Let's create a simple button that changes a text on the Rust side:

// main.rs
use alakit::AlakitEngine;
use rust_embed::RustEmbed;

#[derive(RustEmbed)]
#[folder = "ui/"]
struct Assets;

fn main() {
    AlakitEngine::new("Hello Alakit")
        .with_assets::<Assets>()
        .run();
}
<alakit-meta controllers="hello" />
<button alakit-cmd="hello:greet">Click here!</button>

3. Core Concepts

The Alakit engine is built on a unique messaging bridge (The Bridge).

The Bridge (IPC)

When you click on an HTML element, the value of alakit-cmd is converted into an IPC (Inter-Process Communication) message by the system. It follows the controller:command|args format and arrives immediately in the Rust handle method.

Namespaces

Tip: Organize your controllers by namespaces (e.g., auth, vault, settings). Multiple namespaces can coexist on a single HTML page.

Lifecycle

1. Startup: The AlakitEngine initializes the event loop and the Webview.
2. Discovery: The Rust register (inventory) maps out the structs marked with the #[alakit_controller] macro.
3. Runtime: The Store sends events, and the DOM updates itself.

4. Frontend: Declarative HTML

Alakit's HTML language enables JS-free interactions through special attributes.

Detailed Attribute Reference

alakit-cmd="namespace:command"

Specifies which Rust controller and command to send the message to.

alakit-event="event[.modifier]"

Specifies which DOM event should trigger the IPC message. Default: click.

Type / Modifier Example Description
click alakit-event="click" Default. Executes on click (or touch).
keyup.enter alakit-event="keyup.enter" Only executes when the Enter key is released.
keyup.esc alakit-event="keyup.esc" Only executes when the Escape key is pressed.
input alakit-event="input" Executes on every character typed (or deleted).
change alakit-event="change" Only executes when the element loses focus and its value has changed.
mouseover alakit-event="mouseover" When the mouse enters the element area.

alakit-form="container_id"

If specified, Alakit gathers all input/select/textarea data from within the container with the given ID and sends it to the Rust side in JSON format.

  • Usage: Perfect for complex forms or search bars.

alakit-args="static_data"

Passes a static string parameter to the command. If the element is an input and there's no alakit-args, Alakit automatically sends the current value of the element.

alakit-bind="key"

Two-way binding. If the value of key changes in the Rust Store, all elements with this attribute will update.

alakit-html

Modifier for alakit-bind. If present, Alakit renders the data as raw HTML (innerHTML) instead of safe text (innerText).

alakit-show / alakit-hide

Conditional rendering. The visibility of the element depends on the "truthy" state of the Store value.

Manual JavaScript Call

If you want to message Rust from custom JS logic, use the global function:

alakit("namespace:command|argument");

This sends the message directly to the IPC channel.

5. Backend: Rust Controllers

This is where the real work happens. Controllers are responsible for processing requests.

#[alakit_controller]

Every controller must implement the AlakitController trait and use the registration macro:

#[alakit_controller("vault")]
#[derive(Default)]
pub struct VaultController;

impl AlakitController for VaultController {
    fn handle(&self, command: &str, args: &str, ctx: &AppContext) {
        match command {
            "unlock" => { /* Logic here */ },
            _ => println!("Unknown command"),
        }
    }
}

Advanced Pattern: Auto-Discovery

In larger projects with dozens of controllers, it can be tedious to manually list every file in mod.rs. The Vault app demonstrates a professional solution:

How does it work?
  1. Crfeate a build.rs file that scans the src/controllers folder before compilation.
  2. The script generates a temporary Rust file containing all found modules.
  3. In mod.rs, we only need to use a single line: include!(concat!(env!("OUT_DIR"), "/generated_controllers.rs"));

This way, you just need to create a new .rs file in the folder, and Alakit will automatically register it during the next build.

6. Data Management: The Encrypted Store

This is the heart of Alakit. A central, encrypted memory store that keeps Rust and the UI in sync.

Security and Memory Protection

The Store doesn't store plain strings, but encrypts every value using AES-256-GCM in memory. Alakit pays extra attention to RAM-level protection:

  • Mlock (Memory Locking): Encryption keys are locked in physical memory (via region 3.0), preventing the OS from writing them to the swap file (disk).
  • Zeroize: All sensitive data (keys, temporary buffers, decrypted values) are automatically overwritten with zeros in memory as soon as they are no longer needed.
⚠️ IMPORTANT SECURITY WARNING:

The protection mechanisms mentioned above (encryption, mlock, zeroize) apply exclusively to the Rust-side storage. When data is sent to the frontend (Webview), it resides there in plaintext within the DOM. Therefore, it is critical not to send sensitive information to the UI unless absolutely necessary, as scripts within the Webview can access it.

Secure RAM API Reference

You can access the Store API directly via the ctx.store object in Rust controllers.

Method Arguments Description
set(key, val) &str, &str Encodes and saves the value to RAM and immediately updates the UI on all bound elements.
get(key) &str Returns the decrypted value from memory (Option<String>). Returns None if data is corrupted or doesn't exist.
remove(key) &str Permanently deletes the data from memory and notifies the UI to reset the element.
Mechanism: When set is called, Alakit sends an asynchronous IPC message to the Webview, so the UI update doesn't block the Rust business logic.

7. RustDOM API Reference

The RustDOM API allows direct UI manipulation from the handle method of Rust controllers, without relying on the Store.

RustDOM (Global Operations)

These methods are available on the ctx.dom object.

Method Arguments Description
get_id(id) &str Returns an Element instance to control the HTML element with the given ID.
toast_success(msg) &str Sends a green success notification to the UI.
toast_error(msg) &str Sends a red error notification.
toast_info(msg) &str Sends a blue info notification.
log(msg) &str Sends a development message to the browser console.

Element (Element-specific Operations)

These methods are available on the object returned by the ctx.dom.get_id("...") call.

Method Arguments Description
set_text(text) &str Modifies the element's inner text (safe).
set_html(html) &str Entirely overwrites the element's inner HTML content.
set_value(val) &str Modifies the value of an input or textarea.
add_class(name) &str Adds a CSS class to the element.
remove_class(name) &str Removes a CSS class.
set_style(p, v) &str, &str Directly modifies inline CSS (e.g., "color", "red").
set_attribute(n, v) &str, &str Sets an arbitrary HTML attribute (e.g., "disabled", "true").
remove() - Permanently removes the element from the DOM.

8. Packaging and Distribution

Alakit's goal is Single Binary distribution: one .exe file containing everything.

Optimization

Use rust-embed to embed UI files, so you don't need to copy folders alongside the installer.

cargo build --release

Windows Protection

For Wry-based applications, code signing is important, otherwise Windows SmartScreen might throw a warning. During development, you can bypass this with "More Info -> Run Anyway".

9. Debugging and Diagnostics

One of Alakit's biggest advantages is "Remote-to-Terminal" logging, allowing you to see JavaScript events directly in the Rust console.

Console Redirect

You don't need to open browser DevTools, as Alakit automatically redirects JS logs:

  • console.log() → 🌐 [JS LOG]
  • console.warn() → 🟡 [JS WARN]
  • console.error() → 🔴 [JS ERROR]

Global Debugging

If an unexpected error occurs in frontend code (e.g., typo or null reference), the bridge catches it immediately and displays the exact error message and source file location in the Rust terminal.

Internal system: Alakit uses the alakit:log IPC channel for these messages, so they don't interfere with your custom controller logic.
